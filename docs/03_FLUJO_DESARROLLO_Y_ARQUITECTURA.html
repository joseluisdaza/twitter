🔄 Flujo de desarrollo

El principio fundamental del proyecto es que los modelos Smithy son la única fuente de verdad del API. Los tipos TypeScript, las validaciones y las interfaces se derivan automáticamente a partir de ellos.

flowchart LR A(["📝 1\nEditar\n.smithy"]) B(["⚙️ 2\nGradle Build\n./gradlew build"]) C(["📦 3\nmodels_0.ts\nregenerado"]) D(["🧑‍💻 4\nAdaptar handler\nsi es necesario"]) E(["🚀 5\nnpx cdk deploy"]) F(["☁️ 6\nLambda\nactualizado"]) A --> B --> C --> D --> E --> F style A fill:#4f8ef7,color:#fff,stroke:none style B fill:#7c3aed,color:#fff,stroke:none style C fill:#a78bfa,color:#fff,stroke:none style D fill:#065f46,color:#fff,stroke:none style E fill:#0369a1,color:#fff,stroke:none style F fill:#92400e,color:#fff,stroke:none
1

Editar los modelos Smithy

Los archivos .smithy en smithy/model/ definen el API — operaciones, estructuras, validaciones y errores. Son independientes de cualquier lenguaje de programación.

2

Regenerar código TypeScript con Gradle

El compilador de Smithy lee los .smithy y regenera models_0.ts con los tipos TypeScript y los métodos .validate() para cada estructura.

3

Adaptar el Lambda handler (si es necesario)

Si agregaste campos nuevos o una operación completamente nueva, el handler TypeScript debe actualizarse para usar esos cambios. Si solo cambiaste una restricción, .validate() se actualiza solo.

4

Desplegar con CDK

npx cdk deploy usa esbuild para empaquetar cada handler y solo sube a AWS los Lambdas que cambiaron (comparación por hash).

Ejemplo: agregar un campo nuevo

Para agregar tags a CreateChirpInput en chirps.smithy:

structure CreateChirpInput {
    @required
    content: ChirpContent

    mediaUrls: MediaUrlList

    // Nuevo campo (opcional)
    tags: TagList
}

list TagList {
    member: String
}

⚙️ Etapa 2 — Regenerar código TypeScript con Gradle

Directorio: smithy/

Comando

# Windows
.\gradlew.bat build

# Linux / Mac
./gradlew build

Si hay errores en el modelo Smithy, el build falla con mensajes descriptivos antes de tocar ningún archivo TypeScript.

# Limpiar y recompilar desde cero
.\gradlew.bat clean build

Qué se regenera

smithy/generated/source/
  typescript-server-codegen/src/
    models/
      models_0.ts  ← tipos + .validate()
    server/
      operations/  ← interfaces de operación

Los handlers importan directamente desde models_0.ts, por lo que el nuevo tipo estará disponible en el siguiente cdk deploy sin copiar nada manualmente.

🧑‍💻 Etapa 3 — Adaptar el Lambda handler

Directorio: infrastructure/lambda/handlers/

Los handlers importan los tipos y validadores directamente desde el código generado por Smithy:

import { CreateChirpInput } from '../../../../smithy/generated/source/typescript-server-codegen/src/models/models_0';

export const handler = async (event) => {
  const input = parseBody<CreateChirpInput>(event.body);

  // Validación automática — generada por Smithy
  const failures = CreateChirpInput.validate(input);
  if (failures.length > 0) return badRequest('Validation error', failures);

  // ... lógica de negocio
};

¿Cuándo hay que editar el handler?

Tipo de cambio en Smithy¿Editar handler?Motivo
Nueva restricción en campo existente (ej: @length) No .validate() se actualiza automáticamente
Nuevo campo opcional en una estructura Opcional Solo si quieres usar ese campo en la lógica
Nuevo campo requerido en una estructura Sí TypeScript marcará el error de compilación
Nueva operación completa (nuevo endpoint) Sí Crear handler + registrar en CDK

Si agregas una operación completamente nueva

A

Crear el handler

infrastructure/lambda/handlers/<grupo>/<nombreOperacion>.ts

B

Registrar la función Lambda

Agregar def('nombreOperacion', '<grupo>/<nombreOperacion>.ts') en infrastructure/lib/constructs/chirp-lambdas.ts y asignarle permisos DynamoDB.

C

Agregar la ruta en API Gateway

Agregar this.api.addRoutes({ path: '...', ... }) en infrastructure/lib/constructs/chirp-api.ts.

🚀 Etapa 4 — Deploy con CDK

Directorio: infrastructure/

Comandos de deploy

# Primera vez en una cuenta nueva (una vez por cuenta/región)
npx cdk bootstrap aws://ACCOUNT_ID/REGION

# Deploy completo
npx cdk deploy

# Ver qué cambiaría sin desplegar
npx cdk diff

# Verificar que el template se genera sin errores
npx cdk synth

# Destruir todos los recursos de AWS
npx cdk destroy

Qué ocurre internamente

  1. cdk synth genera el template CloudFormation
  2. esbuild bundlea cada handler en un index.js minificado
  3. CDK compara hashes de los zips con AWS
  4. Solo sube los Lambdas que cambiaron
  5. CloudFormation actualiza las funciones Lambda

⚠️ Nota para Windows

Si el primer intento falla con EPERM: operation not permitted, vuelve a ejecutar el mismo comando. El antivirus escanea los archivos recién instalados y los libera al segundo intento. En Linux/Mac esto no ocurre.
