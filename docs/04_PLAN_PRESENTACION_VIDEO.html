Moisés — Introducción y Requerimientos

Abre el video · establece el contexto general
⏱ ~5 min
🎯 ¿Qué es Chirp? 1 min
  • Red social estilo Twitter construida desde cero con tecnologías AWS
  • Proyecto académico que demuestra arquitectura serverless real en producción
  • 5 desarrolladores · 3 repositorios Git · infraestructura 100% como código
📋 Requerimientos funcionales 2 min
  • Autenticación: registro, inicio y cierre de sesión con JWT
  • Chirps: publicar, leer, eliminar, ocultar, dar y quitar "me gusta"
  • Comentarios: crear y eliminar comentarios en chirps
  • Social: seguir/dejar de seguir usuarios, ver seguidores y seguidos
  • Timeline: feed personalizado con chirps de los usuarios que sigues
  • Perfiles: ver y editar información de usuario
🏗️ Requerimientos no funcionales 2 min
  • Escalabilidad: serverless — escala a cero, paga lo que se usa
  • Seguridad: principio de menor privilegio con IAM, HTTPS en todos los endpoints
  • Type safety: contrato de API verificado en tiempo de compilación con Smithy
  • Automatización: despliegue automático vía CI/CD, sin intervención manual en AWS

🗂️ Estructura multi-repo con Git Submodules 2 min

  • Twitter-core (submodule smithy/) — modelos Smithy, fuente de verdad de la API

☁️ Arquitectura AWS 3 min
  • S3 + CloudFront: frontend estático servido globalmente con CDN
  • API Gateway HTTP API: entry point de las 23 operaciones REST
  • Lambda (Node.js 24): una función por operación — aisladas e independientes
  • DynamoDB: tablas por dominio (chirps, users, follows, comments) con GSIs
  • CDK / CloudFormation: infraestructura declarativa desplegada automáticamente
Usuario → CloudFront → S3 (frontend HTML/CSS/JS)
Usuario → API Gateway → Lambda → DynamoDB
📦 CDK Constructs 2 min
  • ChirpFrontend: S3 bucket + CloudFront OAC + BucketDeployment automático
  • ChirpLambdas: función Lambda por operación, bundleada con esbuild
  • ChirpDatabase: tablas DynamoDB con índices secundarios globales
  • ChirpApi: HTTP API Gateway con rutas mapeadas a cada Lambda

José Luis — AWS Base y DynamoDB / IAM

Configuración de los servicios fundacionales de AWS
⏱ ~7 min
🗄️ Tablas DynamoDB 3 min
  • chirp-users: PK = userId · GSI por username (lookup por username)
  • chirp-chirps: PK = chirpId · GSI por userId (chirps de un usuario)
  • chirp-follows: PK = followerId, SK = followeeId · GSI inverso
  • chirp-comments: PK = commentId · GSI por chirpId (comentarios de un chirp)
  • chirp-likes: PK = userId, SK = chirpId (relación muchos a muchos)
💡 Por qué NoSQL para este caso
  1. Las consultas más frecuentes son por ID — O(1) con PK directa
  2. Los GSIs permiten consultas por usuario sin hacer scan de toda la tabla
  3. DynamoDB escala automáticamente con la carga — sin administración de servidor
🔐 IAM — Principio de menor privilegio 2 min
  • Cada Lambda tiene su propio rol IAM con solo los permisos que necesita
  • createChirp Lambda → solo dynamodb:PutItem en tabla chirp-chirps
  • getTimeline Lambda → solo dynamodb:Query en tabla chirp-follows + chirp-chirps
  • El rol de GitHub Actions (OIDC) → permisos de CloudFormation + S3 + Lambda para deployar
  • Ningún acceso hardcodeado — todo se maneja con AssumeRole en tiempo de ejecución
🔑 Autenticación con JWT 2 min
  • Register: hashea contraseña con bcrypt → guarda en chirp-users
  • Login: verifica hash → genera JWT firmado con secret → devuelve token
  • Cada Lambda protegida verifica el JWT en el header Authorization: Bearer
  • El userId se extrae del token — el cliente nunca envía su ID directamente

Jorge — Funcionamiento y Smithy

🧩 ¿Qué es Smithy y por qué lo usamos? 2 min
Frase de apertura: "Antes de ver cómo funciona el sistema, necesito explicar la pieza que todo lo conecta: Smithy."
  • Smithy es un lenguaje de definición de APIs creado por AWS — similar a OpenAPI
  • Es la fuente de verdad del proyecto: si un endpoint existe, primero existe en Smithy
  • Lo que definimos en Smithy se genera automáticamente en TypeScript, Java, Python, etc.
  • No es documentación — es código ejecutable que se usa en producción

🔄 Flujo completo de una request 3 min
Caso de uso: "El usuario escribe 'Hola mundo' y presiona Chirpear."
1. Usuario → POST /chirps  (Header: Authorization: Bearer [JWT])
2. API Gateway → verifica ruta → invoca Lambda createChirp
3. Lambda createChirp:
     → verifica JWT → extrae userId
     → genera chirpId (UUID)
     → PutItem en DynamoDB tabla chirp-chirps
     → retorna { chirpId }
4. Frontend recibe { chirpId }
     → llama loadTimeline() → actualiza la UI

  2. Escribir un mensaje y presionar "Chirpear"

✨ Lo que hace especial a este sistema 1 min
  • Type safety transversal: Smithy → TypeScript → compilación → CI. Un breaking change se detecta antes de producción
  • Zero servers: cada Lambda vive ~50ms, cobra lo que usa, escala automáticamente
  • Git como fuente de verdad: cada cambio pasa por PR → CI valida → CDK despliega. Nadie toca la consola AWS a mano

Mauricio — Pipelines CI/CD y Conclusiones

Automatización, calidad y cierre del video
⏱ ~7 min
🔁 Pipeline CI — ci_validate.yaml 2 min
  • Trigger: cualquier push o Pull Request a main
  • Job 1 — smithy-validation: corre ./gradlew clean build, valida los modelos Smithy
  • Job 2 — frontend-typecheck: necesita Job 1 → instala deps → tsc --noEmit
  • Job 3 — infrastructure-build-test: necesita Job 1 → npm test con cobertura → cdk synthesize
  • Job 4 — documentation-check: verifica que los archivos HTML de docs sean válidos
smithy-validation
      ├── frontend-typecheck
      └── infrastructure-build-test
documentation-check  (independiente)
🚀 Pipeline CD — cd_deploy.yaml 2 min
  • Trigger: solo en push a main (no en PRs)
  • Asume rol IAM en AWS usando OIDC — sin credenciales hardcodeadas en secrets
  • Genera código Smithy → instala deps → cdk deploy --require-approval never
  • CDK detecta cambios incrementales — solo actualiza lo que cambió
  • Si frontend/dist/ no existe, el CDK construct lo construye automáticamente
🖥️ Mostrar en pantalla: GitHub Actions
  1. Abrir un run exitoso del pipeline en GitHub
  2. Mostrar los 4 jobs del CI en verde
  3. Mostrar el CD con el paso de cdk deploy completado
📈 Recomendaciones técnicas 2 min
  • Usar OIDC en lugar de access keys: credenciales AWS generadas por request, no almacenadas en secrets
  • CDK self-building constructs: la infraestructura sabe cómo construir sus assets — menos dependencias entre pasos
  • Smithy como contrato compartido: cualquier equipo genera su cliente en el lenguaje que prefiera
  • Separar CI de CD: validar en cada PR y desplegar solo en merge — evita deployar código roto
  • Git Submodules para multi-repo: cada equipo trabaja en su repo con historial independiente
🏁 Conclusiones y cierre 1 min
  • Chirp demuestra que es posible construir una plataforma social completa y escalable con servicios AWS administrados
  • El stack (Smithy + CDK + Lambda + DynamoDB) permite iterar rápido sin preocuparse por servidores
  • La inversión en type safety y CI/CD desde el día uno paga dividendos: errores detectados antes de producción
  • Todo el código e infraestructura están en GitHub — reproducible desde cero con un git clone
